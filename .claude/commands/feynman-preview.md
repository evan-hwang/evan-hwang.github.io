# 파인만 학습 블로그 미리보기

현재 세션의 파인만 학습 내용을 **완성도 있는 기술 블로그 글**로 변환하여 미리보기합니다.
**파일은 저장되지 않습니다.**

## 핵심 원칙

파인만 학습 과정(Q&A, 점수, 비유 평가 등)은 **독자에게 보여주지 않는다.**
학습 과정은 입력이고, 블로그 글은 **정제된 출력**이다.

참고 스타일:
- Addy Osmani 블로그: 명확한 문제 정의, 근거 기반 설명, 실용적 조언
- Naver D2: 기술적 깊이, 아키텍처 다이어그램, 컴포넌트 분석

## 글의 깊이 기준

단순 개념 설명으로 끝나지 않는다. 아래 요소를 포함해야 한다:
- **Why**: 이 기술/개념이 등장한 배경, 어떤 문제를 해결하는가
- **What**: 핵심 개념과 구조 (아키텍처, 컴포넌트)
- **How**: 실제로 어떻게 동작하는가 (코드, 설정, 흐름)
- **Trade-offs**: 장단점, 대안과의 비교

학습 세션에서 깊이가 부족하면, Claude가 추가 리서치를 수행하여 보강한다.

## 시리즈 지원

주제가 넓어 한 편으로 다루기 어려운 경우 시리즈로 구성한다:
- 제목에 `#1`, `#2` 등을 붙인다
- 각 편의 범위를 명확히 정의한다
- 시리즈 첫 글에 전체 로드맵을 안내한다

예시:
- "KEDA 딥다이브 #1: 왜 필요한가, 그리고 아키텍처"
- "KEDA 딥다이브 #2: ScaledObject와 트리거 메커니즘"

## 글쓰기 스타일 가이드

- **톤**: 전문적이지만 읽기 쉬운. 독자와 대화하듯 자연스럽게.
- **구조**: 각 섹션이 독립적으로 읽혀도 이해 가능하게.
- **근거**: 주장에는 근거를 제시. 공식 문서, 소스 코드, 벤치마크 등 인용.
- **코드**: 핵심 동작을 보여주는 최소한의 코드. 장황하지 않게.
- **비유**: 복잡한 개념은 일상적 비유로 한 번 풀어주되, 과하지 않게 1-2회만.

## 섹션 구성 유연성

위 섹션은 기본 템플릿이다. 주제에 따라 유연하게 조정한다:
- 개념 중심 주제: "배경 → 핵심 개념 → 동작 원리 → 마무리"
- 도구/프레임워크: "문제 → 아키텍처 → 컴포넌트 → 실전 → 트레이드오프"
- 비교 분석: "배경 → A vs B 구조 비교 → 성능 → 선택 기준"

## 다이어그램

아키텍처, 컴포넌트 관계, 데이터 흐름 등을 시각화할 때 **excalidraw MCP**를 사용한다.

**다이어그램 삽입 기준:**
- 아키텍처 개요: 컴포넌트 간 관계를 보여줄 때
- 데이터/제어 흐름: 요청-응답 흐름, 이벤트 파이프라인 등
- 비교 구조: Before/After, A vs B 등
- 계층 구조: 레이어, 스택 구성

**미리보기 시:**
- 다이어그램이 들어갈 위치와 내용을 `[다이어그램: 설명]` 형태로 표시
- 예: `[다이어그램: Skills 파일이 Claude에게 로드되는 흐름]`

## 주의사항

- 파인만 학습 과정(Q&A, 점수, 비유 평가)을 그대로 노출하지 않는다
- "나의 설명", "피드백" 같은 학습 일지 형식을 사용하지 않는다
- 학습에서 발견한 인사이트는 자연스럽게 본문에 녹인다
- 깊이가 부족한 부분은 Claude가 추가 리서치하여 보강한다
